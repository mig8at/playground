# Motai — flujo de originación (v1, como ES hoy)

Flujo de originación del comercio Motai (allied 158) con su lender MotaiX (id 158, response_type=2, CreditopX in-platform; el flag isMotai = MOTAI_LENDER_IDS.includes(id), MOTAI_LENDER_IDS=[158]). Motai es un COMERCIO con 3 MODOS (compra 'motai' / renting 'motai-renting' / alquiler 'alquiler') persistidos en allied_modes/user_request_modes; lo distintivo del flujo es el MODO + ABACO (scraping de ingresos gig, SOLO en renting via isAbacoRequired, que salta Experian/buro). Recorrido: application da de ALTA el lender 158 rt=2 desde el admin (fila runtime sin seeder): crea el lender + su CreditopXLenderConfiguration, lo adjunta al comercio/sucursal y copia reglas + datacredito por sucursal. Onboarding: monto+telefono -> OTP (transporta isMotaiRenting) -> datos personales/laborales (con el bypass MOTAI/PEP: info laboral dummy, la validacion de ingresos se hace luego via Abaco) + seleccion del MODO. Si renting: gate Abaco (platforms -> init gig-economy -> login+OTP -> results, mock en local; persiste en UserSummary.abaco, informativo/no cableado hoy). Marketplace lenders-v2 y seleccion del lender 158 (branch isMotai en LenderCardContent; persiste lender_id + user_request_status_id 3). Cierre CreditopX estandar: confirm -> biometrica ADO + AML Tusdatos -> fecha primer pago -> plan de cuotas -> firma pagare + OTP -> authorize -> Estado 11 (LoanAuthorizationService::authorize; estados DB-driven en user_request_statuses, el 11 hardcodeado). Callback de servicing motai/update-status. Incluye además, para cerrar las brechas de Alta/MVP2 (análisis de brechas): el subsistema legal/TyC hoy acoplado a Credifamilia (LegalService + AcceptedTermsAndConditionsS3Repository + SignedTermsAndConditionsMail + el enum document-type + otp-resend), el MOTOR DE VIABILIDAD que Motai SALTEA hoy (DatacreditoQueryByAlliedController::userViability, base para la política de riesgo R1-R8 de MVP2) y el perfil financiero que pasa al administrador (customer-profile/financial-profile).
