# CreditOp — contexto del ecosistema

CONTEXTO RAÍZ de CreditOp: qué es y cómo se reparte la lógica en los 3 repos. Este nodo es el PADRE de todos los flujos (Motai, etc.) — leélo primero para entender el ecosistema antes de bajar a un flujo puntual.
QUÉ ES. CreditOp es una plataforma colombiana de crédito en punto de venta y ecommerce que conecta 3 actores: COMERCIOS aliados (allieds, con SUCURSALES/allied_branches como puerta de entrada por hash), CLIENTES que financian una compra, y ~150 LENDERS que ponen el dinero. Usa DOS SOMBREROS: (1) BRÓKER/marketplace — muestra opciones y arma la solicitud pero un tercero externo presta, decide y cobra (response_type 0 UTM/referido, 1 Integración por API, 4 Credifamilia por SOAP); y (2) OPERADOR bajo la marca CreditopX (rt=2 in-platform, rt=3 cupo rotativo) — CreditOp origina, firma con OTP+pagaré, desembolsa hasta el Estado 11 (Autorizada) y lleva la cobranza, PERO el capital y el riesgo los pone el COMERCIO (marca blanca; CreditOp gana comisión por recaudo). Una operación = combinación de 4 ejes ortogonales: response_type (quién decide/gestiona), producto/garantía (compra · SmartPay=celular con bloqueo MDM · Motai=arrendamiento/renting), modo del comercio, y canal (WooCommerce/self/VTEX, asesor con QR, Corbeta por lotes). El Estado 11 es la frontera entre originación y servicing/cartera.
LOS 3 REPOS (migración strangler-fig en parallel-run). La lógica vive repartida entre application (el VIEJO) y el par legacy-backend + frontend-monorepo (el NUEVO). En NEGOCIO: a application se le dice ALIADOS; al conjunto legacy-backend + frontend-monorepo se le dice REFACTOR.
ALIADOS = application. Monolito Laravel+Vue (NO modular): dominio en app/ (Models, Http/Controllers/{Admin,Customer,Api}, Services, Console/Commands, Jobs) y rutas por audiencia (admin.php, customer.php ~37KB, api.php). Es el sistema HISTÓRICO y el runtime POR DEFECTO: sigue corriendo la originación de casi todos los comercios, y aloja EN EXCLUSIVA la creación de ENTIDADES/COMERCIOS, la asignación de SUCURSALES (con la copia de reglas/datacrédito por sucursal), el panel admin + SSO Cognito de aliados, y TODA la cobranza/servicing post-desembolso (crons diarios de mora, pagos, revolving = UpdateCreditopX*).
REFACTOR = legacy-backend + frontend-monorepo. Es el DESTINO de la migración; ya reconstruyó el núcleo de originación CreditopX (cierre rt=2/3, ecommerce/VTEX con contrato base64, cuota inicial Wompi, identidad, device-lock SmartPay), self-contained y validado e2e. legacy-backend = Laravel MODULAR (nwidart/laravel-modules, Modules/* por dominio: Onboarding · Loans · Risk · Identity · Payments · Partner · System, más la nueva arquitectura V1/V2 = Controller delgado -> Service -> Command -> Repository con envelope {code,message,data}). frontend-monorepo = monorepo Turborepo+pnpm (apps/loan-request-wizard = wizard React Router v7 + SSR; packages/* = librerías @creditop/*; modules/loan-request-wizard/* = dominios DDD domain/application/infrastructure/ports/ui). El frontend NO toca la BD: es puro cliente HTTP de legacy-backend (VITE_API_URL).
BASE DE DATOS COMPARTIDA. application y legacy-backend comparten LA MISMA base de datos (misma RDS/Redis/Pusher en prod durante el parallel-run). Por eso app/Models/ se REPITE en ambos (User, Allied, AlliedBranch, Lender, LendersByAllied, LendersByAlliedBranch, UserRequest y todo el ledger CreditopX*): el puente REAL application<->legacy es la BD, no HTTP. Dos ejes que NO hay que confundir: DEPENDENCIA (legacy NO llama a application en runtime — 0 dependencias de código; el único acople de código es application->legacy vía GenerateServicesBridgeClient / NewFrontendUrlService) vs CUTOVER (application todavía CORRE porque el ruteo es un allowlist por comercio hardcodeado y creciente, ej WoocommerceController [24,209,210,211,311]; cualquier comercio fuera de la lista corre íntegro en application).
LÓGICA DUPLICADA / SOLO-EN-REFACTOR / FLUJOS QUE CRUZAN (la migración NO terminó). DUPLICADA: el cierre CreditopX (pagaré), VTEX, documentos web y notificaciones existen como copias paralelas vivas en ambos; el listado de lenders (LenderRetrievalService en Aliados ≈ LenderListingService en Refactor), el datacrédito (RiskCentralValidationService viejo ≈ DatacreditoRuleEvaluator nuevo) y el alta de comercios (Admin/Allied* en Aliados ≈ Modules/Partner en legacy, este último GEMELO MUERTO no invocado) reaparecen más ORGANIZADOS en Refactor. SOLO EN REFACTOR: dynamic-forms de SmartPay, notificador de ecommerce unificado. FLUJOS QUE CRUZAN AMBOS: una SOLICITUD se origina en legacy pero continúa su vida (cartera) en application; el callback biométrico ADO postea hacia application; el wizard nuevo hace de BRIDGE hacia el panel viejo (SSO Cognito HMAC a ALIADOS_BASE_URL); los webhooks entrantes de agregadores rt=1 están COPIADOS-SIN-RUTA (código muerto) en legacy y solo funcionan en application. Tesis de fondo: CreditOp se adapta a cada comercio con ifs quemados por ID en vez de configuración; el deber-ser es un modelo único paramétrico con reglas heredadas (no copiadas ~37.000 veces por sucursal).
CÓMO LEER LOS ARCHIVOS (abajo, por repo). application/: rutas por audiencia (admin/customer/api), los modelos de la BD compartida, el god-service del listado (LenderRetrievalService), el alta de comercios/sucursales (Admin/Allied*), el cron de servicing (UpdateCreditopXRequests) y los 2 puentes cross-repo + config/services.php. legacy-backend/: bootstrap + config de módulos, los MISMOS modelos de la BD compartida, el módulo Onboarding (originación migrada: listado + perfilamiento), el módulo Loans (datacrédito nuevo, autorización=Estado 11, pagaré, flujo CreditopX), Identity (risk centrals/buró) y Partner (gemelo muerto del alta). frontend-monorepo/: el mapa de rutas RRv7, el root SSR, el contrato de env (VITE_API_URL + ALIADOS_BASE_URL + SSO), el HttpClient canónico y los repositories/módulos DDD (marketplace, originación, backend-driven-form, Ábaco) que pegan a legacy.
