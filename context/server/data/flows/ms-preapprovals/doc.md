# MS Pre-approvals · contexto
> **estado:** al día con main · MS Go hexagonal (`pre-approvals-service`, repo aparte NO indexado en este árbol). Resuelve la pre-aprobación de lenders rt≠0 (principalmente rt=1) del wizard nuevo contra las APIs externas, vía un workflow de 4 etapas, cachea en DynamoDB y espeja el resultado a legacy.

## Qué es
Microservicio **Go** de arquitectura **hexagonal estricta** (`pre-approvals-service`, módulo `github.com/Creditop-SAS/go-template`; Go 1.24 · Gin · Uber FX · Viper · DynamoDB · `shopspring/decimal`; linter `depguard`). Resuelve la **pre-aprobación de lenders de integración (rt≠0, principalmente rt=1)** para el wizard nuevo: recibe un `POST /v1/preapprovals/check` **por lender**, arma el payload, golpea la **API externa del proveedor** (Welli, Meddipay, Prami, Bancolombia, Sistecrédito…), canoniza el veredicto a `approved`/`rejected`/`pending`, lo **cachea en DynamoDB** y, best-effort, **espeja el resultado terminal a legacy** para el perfilamiento. Repo aparte (`github/pre-approvals-service`) — **NO está en el índice de los 3 repos** de este árbol; sus anclas van como conocimiento (ver §Contenido).

Coexiste con el path viejo (`PreApprovedLenderService` PHP en application/legacy) en **parallel-run**: el wizard nuevo consume este MS directamente; el monolito sigue con su switch por id. Los dos mundos pueden divergir. El detalle a nivel de **flujo rt=1** (familia de lenders, handoff/entrega, Corbeta, webhooks de cierre) es del nodo hermano **Aggregator**; acá se describe el **servicio** (la pieza de arquitectura).

**Frontera de responsabilidad (rt=1):** CreditOp solo (a) decide **qué** lenders consultar, (b) aporta **datos** del solicitante (Experian/KYC vía customer-service) y **credenciales** del comercio, (c) **traduce** la respuesta y **ordena**. La **decisión de crédito, monto y cupo la calcula 100% la API externa** — por eso rt=1 **NO es inyectable/simulable** de punta a punta (a lo sumo se mockea el transporte HTTP).

## Contenido
- **Contrato HTTP** — `POST /v1/preapprovals/check` (server-to-server, sin CORS; también `/me/check` con header `X-User-Id`; rutas en `handler.go:252`, `Check` en `:45`, `CheckMe` en `:146`). **Request**: `applicant_id`, `merchant_id`, `lending_product_key` (enum), `lending_product_id` (numérico), `amount`/`allied_branch_hash`/`allied_branch_id` (opcionales, requeridos por algunos lenders), `user_request_id` (opcional → dispara intento + notify a legacy). **Response 200**: `status`, `approved_amount`, `available` (cupo), `probability`/`probability_color`/`sort`, `transaction_id`, `transaction_data` (blob por-lender), `checked_at`. `expires_at` **NO viaja** (es interno de DynamoDB para frescura/TTL). **Errores**: `400` (JSON/campo faltante o `invalid lending product key`), `422` `amount_below_minimum` + `minimum_amount` (`handler.go:124`), `500` (típicamente `transport_error` del proveedor: timeout/unavailable).
- **Workflow de 4 etapas** — `workflow.go:52 CheckPreApproval`, idéntico para todo lender: `runCredentials(:101)` → `runAuth(:117)` → `runAPICall(:131)` → `runAdapt(:145)`; cada etapa abre un span OTel y aborta con `failStage(:167)`. Cada lender aporta 3 piezas intercambiables: **CredentialsService** (scope), **AuthStrategy** y **Adapter**.
- **Factory / 8 productos** — `factory/factory.go` mapea la key a `client + authStrategy + adapter + CredentialScope`. Las **8 keys**: `bancolombia_bnpl`, `bancolombia_consumer_loan`, `sistecredito`, `meddipay`, `creditop_x`, `welli`, `prami`, `credifamilia`. Key desconocida (ej. `banco-de-bogota`) → `400 invalid lending product key`. **CredentialScope**: **Merchant** = Bancolombia BNPL/Consumo, Meddipay (una credencial por comercio); **Branch** = Sistecrédito, CreditopX, Welli, Prami, Credifamilia (por sucursal → se reenvía el `allied_branch_hash`).
- **Adapter (canon del veredicto)** — traduce la respuesta cruda a `approved`/`rejected`/`pending` — el **core enum son SOLO esos 3** (`domain/lending_product.go:79`). Ej. Welli (`welli/adapter.go:62`): `approved` solo si `estado=='approved' && monto_aprobado>0` (`:71`/`:112`); `available=monto_maximo_credito`. `BuildContractFields` (`contract_fields.go:20`) fija el contrato con el front: approved → `'Pre aprobado'`/`text-success`/`sort=1`/`pre_approved_lender=true`; else → `'Rechazado'`/`text-info`/`sort=2`.
- **Use case** — `check_preapproval.go:56 Execute`: coerción amount/branch → **cache lookup** (`ShouldCheckAgain`, `:81`) → `GetApplicant` (legacy `…/api/onboarding/user/{id}`, trae `ExperianProfile`) → **override Welli 141/142 → '23'** (`:122`; **166 NO se colapsa**) → workflow → `Save`/`Replace` en DynamoDB + registra `lender attempt` si hay `user_request_id` → `notifyLenderResult` (`:160`/`:199`).
- **Cache DynamoDB** — tablas `preapprovals-dev` + `preapproval_attempts-dev` (region `us-east-2`). `ShouldCheckAgain` (`preapproval.go:42`): **Meddipay siempre re-consulta** (`:45`, necesita nuevo `order_id`); `pending` siempre (`:49`); `rejected` si cambió el monto o pasó `RejectedRetryHours=12h` (`:53`); `approved` válido si `ExpiresAt` es futuro (`:61`). El intento es idempotente por `user_request_id+lender_id+allied_branch_id`.
- **Notify a legacy** — `lender_result_service.go:41 NotifyLenderResult` = `POST {legacy}/api/onboarding/loan-application/{ur}/lender-result` con `{lender_id, is_approved, available_amount}`. Guardas (`check_preapproval.go:199`): **solo** si hay `user_request_id` **Y** el status es `approved`|`rejected` (**`pending` y sin-UR se saltan**); best-effort (falla = log, no bloquea la respuesta). En legacy lo recibe `ListLenderController::storeLenderResult` (`:88`) que repuebla `profiling_reviews.displayed_lenders` vía `updateAsyncLender`; ruta **backend-to-backend sin Cognito**, `withoutMiddleware(AddOriginationFlowType)` (`webhooks.php:18-20`).
- **Config / puertos** — **8082** = address de red en dev/prod (`http://pre-approvals-service.inertia-develop:8082`, lo resuelve `VITE_PREAPPROVALS_ENDPOINT`); **8086** = bind local del `http_server` (`config.example.yaml:24`) — red (deploy) vs bind (local), no es contradicción. `read/write_timeout=30s` (`:25`); timeout por lender **10s** (`:60`; Credifamilia mayor). `LENDER_PROFILE` selecciona perfil: `lambdas-local` (mocks en `:3001-3003`) / `lambdas-aws` / `lenders-staging` / `lenders-production` (los `.yaml` reales están gitignoreados; ⚠️ prod cuenta a central de riesgo).
- **Consumo desde el wizard** — el loader de `available-lenders.tsx` lee `VITE_PREAPPROVALS_ENDPOINT` (`:147`) y **sin endpoint / `user_id` / `allied_id` salta TODO el bloque** (`:185`) → todos los rt≠0 quedan "No pudimos consultar esta entidad". Dispara una `Promise` por lender: los no-fallback en paralelo, los `is_fallback_lender` **esperan al batch primario** (`Promise.allSettled`, `:235`), y Welli usa un consult-lender especial (`:206-226`). El cliente `fetch-lender-preapproval.ts` hace `POST` (`:174`) y mapea `status` → estado UI: `approved` → card pre-aprobado; `rejected` → chip "Sin cupo disponible" (soft, **no oculta**); `pending` → skeleton (Credifamilia hace polling); error `http_5xx` → **card OCULTA** (`SERVER_ERROR_REASON=/^http_5\d{2}$/`, `:65`; `isServerErrorResolution` en `visibleLoanOptions`, `AvailableLenders.tsx:435-441`); error `http_4xx` (negocio) → "No pudimos consultar" + **Reintentar** (`preapproval-retry.tsx` re-golpea el MS). El `422` se mapea a `below_minimum`+`minimumAmount` (`:184`). El front postea `visibleLoanOptions` (lo que ve el usuario) de vuelta a legacy vía `/lender-results` **plural** (`storeLenderResults`) — relay distinto del notify singular del MS.
- **Contraparte legacy (parallel-run)** — `PreApprovedLenderService::validatePreApproveLender` (`:41`): switch **bifurcado por `lender->id`** (no polimórfico) que consulta la misma API externa y **empuja** a `$approvedLenders` (sort=1) **o excluye** con `unset`. Filtro **temporal** en `LenderRetrievalService.php:252` `[12, 23, 141, 142, 166]` (Prami + variantes Welli + 166; `// TODO: [TEMPORAL]` en `:248`) los saca del preaprobado sincrónico porque erroran por falta de datos en `employment-info`.

## Gotchas / riesgos
- **Dos mundos paralelos que divergen**: p.ej. Welli id **166 solo existe en application**; en legacy/MS es 23/141/142 — y el MS colapsa 141/142 → 23 pero **NO** 166.
- **El espejo a legacy es best-effort**: si `/lender-result` falla se loguea pero no bloquea → posible deriva entre lo que ve el usuario y lo que persiste el profiling. Y **`pending` NO repuebla `displayed_lenders`** (solo `approved`/`rejected` disparan el notify; el `pending` mantiene la misma fila por `Replace` para no romper el polling).
- **Meddipay nunca cachea** (`ShouldCheckAgain=true` siempre → nuevo `order_id` por request).
- **Race Credifamilia deliberado**: el front tolera más que el `write_timeout=30s` del MS a propósito (abortar mid-flight duplicaba transacciones); Credifamilia es el **único async** (el front hace polling del `pending`).
- **Estados que el MS NO emite**: su core enum son 3 (`approved`/`rejected`/`pending`). El front reconoce además `frontend_response` (+`encrypt_code`) — que son de **Bancolombia en el legacy PHP**, no del MS — y `not_eligible` (sin emisor en el core enum; manejo defensivo/forward-looking del front). `transaction_data` es un **blob por-lender sin contrato** (`json.RawMessage`, `contract_fields.go:17`; persistido como string en DynamoDB) que fluye **verbatim** al front.
- **Proveedor externo caído (5xx/timeout) → la card DESAPARECE** del marketplace (no muestra error); rechazo de negocio (4xx) → "No pudimos consultar" + Reintentar. Welli/Meddipay son intermitentes (caídos y luego `approved` en corridas consecutivas).
- **`creditop_x` es una de las 8 keys del factory** (available-quota rt=2 hardcodeada), pero el cupo rt=2 productivo se sella en legacy `/available-quota`; no confundir el nodo (rt=1) con esa key.
